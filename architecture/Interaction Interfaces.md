Интерфейсы взаимодействия

# Назначение

Этот документ описывает разрешённые способы взаимодействия с приложениями Human Horizon.

Здесь фиксируются не библиотеки и не реализации, а внешние точки входа: CLI, HTTP API и WebSocket. Если проект хочет добавить новый способ взаимодействия, это должно быть зафиксировано отдельным решением.

# CLI

CLI — это прямой запуск команды через бинарный файл.

**Использовать для:
**
• одноразовых команд
• скриптов
• CI/CD-задач
• локальных инструментов
• детерминированного выполнения команд

**Плюсы:
**
• быстрый старт с точки зрения пользователя
• нет TCP-рукопожатия
• прямое взаимодействие с процессом
• простая интеграция с CI/CD

**Минусы:
**
• накладные расходы на запуск процесса при каждом вызове
• нет состояния сессии между вызовами
• неэффективно для частых последовательных операций

# HTTP API

HTTP API — это взаимодействие в формате запрос-ответ через HTTP-эндпоинты.

**Использовать для:
**
• операций без состояния
• интеграций
• внешних сервисов
• сценариев запрос-ответ

**Плюсы:
**
• стандартная интеграция в стиле REST/JSON
• переиспользование keep-alive-соединений
• хорошая поддержка существующими инструментами

**Минусы:
**
• накладные расходы на первое соединение
• нужен явный маппинг ошибок
• сессии и состояние должны быть спроектированы намеренно

# WebSocket

WebSocket — это постоянное двунаправленное соединение.

**Использовать для:
**
• обновлений в реальном времени
• потоковой передачи данных
• push-уведомлений
• постоянных сессий
• агентских сессий

**Плюсы:
**
• одно рукопожатие на много сообщений
• низкие накладные расходы после установки соединения
• двунаправленная коммуникация
• поддержка состояния сессии

**Минусы:
**
• нужно управлять жизненным циклом соединения
• нужна стратегия переподключения
• обработка ошибок сложнее, чем в CLI или HTTP API

# Правило

Проект может открывать только те интерфейсы взаимодействия, которые он явно объявляет.

Добавление нового интерфейса взаимодействия требует явного решения.
